# Xiaohongshu Cover Generator

## Overview

根据用户提供的视频脚本，输出可直接执行的小红书封面方案。  
默认生成 `3` 套方案并推荐 `1` 套，默认语言中文，默认画幅 `3:4 (1080x1440)`。

## Use Resources

先读取 `references/output-template.md`，严格按模板输出。  
再读取 `references/cover-playbook.md`，复用其中的封面规律、配色策略与布局语法。  
再读取 `references/category-prompt-templates.md`，根据分类选择示范提示词骨架。  
如需直接产图，再读取 `references/image2-direct-generation.md` 并调用 `scripts/generate_image2_cover.py`（支持 OpenAI / Gemini）。  
需要风格锚点时，优先参考 `assets/examples/` 下的分类目录示例，但不要照搬构图。

## Required Inputs

要求用户至少提供视频脚本文案。  
接受以下可选信息：
- 产品名或功能名
- 目标人群
- 风格偏好
- 参考封面路径
- 是否“需检索”
- 是否“直接生成海报”（默认否）

输入缺失时先做最小追问；如果无法补齐，继续产出并在结果中标注假设。

## Workflow

1. 提炼脚本洞察  
从脚本中提炼三类信息：
- 引爆点：最反常识、最高价值或最快见效的能力点
- 核心能力点：视频真正想证明的产品能力
- 情绪锚点：惊喜、效率、掌控感、专业感等

2. 判定封面分类  
按 `references/category-prompt-templates.md` 的规则先判定分类。  
可选分类固定为：`对比效果`、`情绪表达`、`插画风格`、`真人形象`。  
分类优先级：
- 用户明确指定分类 > 用户风格偏好 > 脚本语义自动判定
- 若无指定，三套方案至少覆盖两种分类

3. 决定是否检索  
默认不检索网络。  
仅当用户明确写出“需检索”或同义明确要求时，执行检索并输出来源链接。  
检索范围固定为：
- 官方发布物料或文档
- 同类自媒体封面案例

4. 生成主副标题  
每套方案均生成主标题和副标题，并执行字数校验：
- 主标题：`6-10` 汉字
- 副标题：`6-10` 汉字
- 计数口径：去空格与标点后，仅统计汉字字符数
- 不合规时自动改写，直到合规

5. 生成三套封面方案  
固定输出 `方案A`、`方案B`、`方案C`。  
每套都必须包含：
- 案例分类
- 主标题
- 副标题
- 视觉构图
- 配色
- 元素
- 排版

6. 推荐一套并说明理由  
给出 `推荐方案`，理由至少覆盖：
- 与脚本能力点的贴合度
- 首屏点击冲击力
- 制作可行性

7. 输出制作提示词  
对每套方案均提供：
- 中文通用版提示词
- Midjourney 英文版提示词

8. 直出海报（可选）  
当用户明确要求“直接生成海报”时，优先使用推荐方案的中文提示词调用：
`python scripts/generate_image2_cover.py --provider ... --prompt-file ... --output ... --model ...`  
执行规则：
- 若已配置对应 key（OpenAI 或 Gemini）：直接执行并返回输出路径
- 若 key 缺失：返回待执行命令和缺失项，不中断策划输出

## Hard Constraints

- 默认尺寸：`3:4`，建议 `1080x1440`
- 不强制出现产品 logo、产品截图或 UI 截图
- 优先使用“脚本创意隐喻表达”驱动画面
- 不输出含糊建议，输出可直接执行的构图与文案
- 直接产图支持双 provider：
  - OpenAI：默认 `gpt-image-1`（兼容 `image2.0` 别名），默认 `1024x1536`
  - Gemini：默认 `gemini-2.5-flash-image`（兼容 `nanobanana` 别名），默认 `3:4`

## Title Counting Rule

按以下规则校验主副标题长度：
1. 删除空格字符  
2. 删除常见中文和英文标点  
3. 仅保留汉字字符并计数  
4. 计数不在 `6-10` 时重写标题

示例校验：
- `7天做爆封面！` -> 有效汉字 `5`（“天做爆封面”）需重写
- `AI一键做封面` -> 有效汉字 `6`（合规）

## Output Contract

必须严格遵循 `references/output-template.md` 的字段顺序与命名。  
禁止省略字段；未知信息使用“基于脚本推断”并写出假设。
每套方案必须标记 `案例分类`，并与其提示词风格保持一致。
若用户开启“直接生成海报”，必须追加：
- 执行命令
- provider 与 model
- 输出路径
- 执行状态（成功/失败）

## Example Intents

- 基于这段脚本生成小红书封面，给我3套并推荐1套
- 需检索：请补充官方与竞品封面参考后再出方案
- 把这组新封面加入案例库并更新风格规则

## Reference Library Update Policy

仅在用户明确要求“加入案例库”或“更新风格规则”时更新参考库。  
没有明确指令时，不自动追加新规则，避免风格漂移。
新增分类示例时，同时更新 `references/category-prompt-templates.md` 的对应模板。
